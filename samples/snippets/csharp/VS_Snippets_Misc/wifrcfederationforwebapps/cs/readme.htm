Federation Passive Scenario with Identity Provider Selection

The purpose of this sample is to show how federation can be set up with passive clients (web browsers) and provide a mechanism for identity provider selection through home realm discovery.

This sample shows a typical federation scenario where an intranet site allows its employees access to partner sites which support federation. The partner’s federation provider is able to communicate with identity providers (either one or more) configured in an intranet site. In addition, the claims issued by identity providers (for example: age claim) are transformed to some other claims (for example: age range claim or over-21 claim) in the federation provider so that it is consumable by a relying party.

 

Depending on the resources selected by a user, they are either redirected to a single identity provider or prompted to select the identity provider they want to use from a drop down list.

Projects included

1.       CorpWebSite: A sample intranet site that offers links to the resources hosted by partner sites.

2.       FederationPassiveRP: A relying party or resource partner which hosts the resources that users want to access.

3.       FPSTS: A Federation Provider Security Token Service that can talk to Identity Provider Security Token Services.

4.       IPSTS1: An Identity Provider STS that expects Windows Authentication and can issue claims such as the myID and myAgeClaim.

5.       IPSTS2: An Identity Provider STS that expects Windows Authentication and can issue claims such as the myID and myZipcodeClaim.

Running the sample

·           Access: https://localhost/CorpWebSite/Default.aspx

·           Click one of the links to access a resource. The first link will authenticate you directly with IPSTS1 and show you the claims that were issued.

·           Sign out of the relying party page by clicking the “Sign Out” link. You will be redirected back to the CorpWebSite.

·           Clicking the second link will take you to a page where a drop down is shown to select either IPSTS1 or IPSTS2. Depending on the selection you make, you will be authenticated against a specific IPSTS.

·           In the relying party page, you can see which IP STS issued the initial claims by looking at the IdentityProviderIdentifier claim in the claims table. It will show either IPSTS1 or IPSTS2. Also, you can notice the differences between the IP and FP STS claims and how the FP-STS has transformed the claims that it got from the IP-STSes. For example, the myAgeClaim from IPSTS1 is transformed to the myAgeRangeClaim at the FP STS.

·           Run the Cleanup.bat to delete the virtual directories created.

User experience flow

·           A user browses to CorpWebSite page and selects a resource to access. For one resource only one Identity Provider (IPSTS1) will be contacted. For the second resource the user needs to select an identity provider to authenticate against.

·           A user selects the first resource. She then gets redirected to the resource site which then redirects to the Federation Provider STS (FP STS).

·           The FP STS detects if there is a whr parameter, if it is then it redirects to the corresponding IP STS (in this case it would be IPSTS1).

·           The IP STS authenticates the user; since windows authentication is enabled you shouldn’t be asked for credentials. The IP STS will issue a token for this authentication with the necessary claims (ID & Age claim) and forwards that to FP STS.

·           The FP STS receives the IP issued claims and transforms them to RP readable claims. This part is where the claims get transformed into claims suitable for relying party consumption. For example, the Age claim (e.g. 25) is transformed into the AgeRangeClaim (e.g. Age is >= 21) at the FP STS so that only the needed information is provided to RP.

·           Upon receiving the token from the FP STS, the RP enumerates all the claims and shows them in the page. Note that for illustrative purposes we are showing the claims there were issued by IP STS as well. These claims have a claimtype that contain IPSTS1 or IPSTS2 somewhere in them.

 

Note

The relying party application(s) in this sample can accept tokens issued by an Active Directory Federation Services 2.0 STS instance instead of the sample STS instances. You would need to modify the STS configuration parameters such as ‘issuer’ in the application’s configuration. Any of the STSs in this sample can be replaced by an AD FS 2.0 STS. You must ensure that the AD FS 2.0 STS configuration parameters are reflected in the remaining STS in the sample, as appropriate. Refer to the AD FS 2.0 product documentation for instructions on how to register a claims provider and a relying party application with an AD FS 2.0 STS instance.
